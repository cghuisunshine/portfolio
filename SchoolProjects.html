How to Play

Create a lobby, share a 6-character code, join with friends, and play turns in real time.

Host + Join
Multiplayer lobby flow
Play Live
  1. Host clicks Host to create a lobby and get a code.
  2. Friends click Join, enter the code, and appear in the lobby.
  3. Each player picks a character.
  4. Host clicks Start to begin the match.
  5. On your turn, roll to move; if a fork appears, choose a path, then end your turn.
  6. If two players land on the same tile, a dice battle triggers to determine the winner.
  7. Reach the end of the board to trigger a boss dice battle.

Build Details

Tech stack, architecture, and the parts I implemented (assets, sync, and gameplay interactions).

Client
Browser game
  • Phaser 3 scenes and UI for menus, lobby, board map, and battle flow.
  • TypeScript for gameplay logic and scene/state structure.
  • Vite dev server/build pipeline.
  • Socket.IO client for lobby + turn updates.
Server
Realtime game services
  • Node + Express with Socket.IO for game rooms and state events.
  • MongoDB for asset storage (small files in a collection; large files streamed via GridFS).
  • Redis caching + preload of critical assets for faster first-load.
  • In-memory active match state for quick server-side simulation.
Architecture
Realtime gameplay + cached asset delivery

  |  Express asset route
  |    -> Redis cache (hot/preloaded assets)
  |    -> MongoDB (small assets) / GridFS (large assets)
  |
  |  Socket.IO (rooms by gameId)
  v
Node/Express + Socket.IO
  - lobby: host/join/start
  - game loop: moves/turns/battles
  - emits snapshots for UI rendering
Realtime Sync
How clients stay consistent
  • Lobby: hostLobby, joinLobby, startGame.
  • State: server broadcasts gameState snapshots for rendering.
  • Turns: movePlayerDiceRoll + fork prompts (pathChoiceRequired / choosePath), then endTurn.
  • Feedback: playerMoved, turnAdvanced, battleStarted (tile collisions).
My Role (Key Contributions)
What I implemented and why it matters
  • Assets pipeline (MongoDB + GridFS): implemented/used a split strategy where small assets come from a MongoDB collection and larger assets stream from GridFS, served through the Express /assets/* route.
  • Redis preload for quick access: preloaded critical assets into Redis on server start to reduce initial loading stalls and speed up repeat fetches.
  • Board state synchronization: kept all clients consistent by emitting server snapshots (gameState) and movement/turn events, with periodic sync to recover from missed updates.
  • Route fork interaction + movement: built the fork/branch flow where a dice move can pause on a fork, prompt the player to choose a route, then continue remaining steps while keeping everyone synchronized.
  • Dice-driven gameplay: implemented dice rolling as the core input for movement and battles (player-vs-player collisions and the boss fight at the end of the board).
  • Demo deployment: deployed and validated the live demo at commandz.gochatus.org:30006.
